# Project Architecture & Bug Notes

## How Payables Are Created for a Vendor

Payables (credits on `vendor.payable_account`) are generated by **more than one
transaction type**. Always account for **all** of them.

| Transaction service | `transaction_type` in JournalEntry | Effect on `vendor.payable_account` |
|---|---|---|
| `transactions.services.vendor_bill` | `vendor_bill` | **Credit** (liability created) |
| `inventory.services.record_inventory_purchase_on_credit` | `inventory_purchase` | **Credit** (liability created) |
| `transactions.services.vendor_refund` | `vendor_refund` | **Credit** (tracked separately – vendor owes us) |
| `transactions.services.vendor_payment_against_bill` | `vendor_payment` | **Debit** (liability reduced) |
| `transactions.services.vendor_advance_settlement` | `vendor_advance_settlement` | **Debit** (liability reduced) |

## Bug Fixed (2026-02-28): Inventory Credit Purchase Not Showing in Vendor Detail

### Symptom
Purchasing inventory on credit from a vendor showed the correct payable balance
in the **vendor list** but showed **zero** in the **vendor detail** page.

### Root Cause
`VendorDetailView.get_context_data()` (file: `vendors/views.py`) calculated
`total_billed` with a hard-coded transaction-type filter:

```python
# WRONG – misses inventory_purchase credits
total_billed = JournalLine.objects.active().filter(
    account=vendor.payable_account,
    journal_entry__transaction_type="vendor_bill",   # ← too narrow
).aggregate(v=Sum("credit"))["v"] or Decimal("0")
```

`record_inventory_purchase_on_credit()` creates entries with
`transaction_type="inventory_purchase"`, not `"vendor_bill"`.  Those credits
were **silently excluded** from `total_billed`, so the outstanding payable in the
detail view was zero.

The vendor **list** was correct because it calls `vendor.balance_payable()` (the
model method), which sums **all** credits on the payable account without any
transaction-type filter.

### Fix
Changed `total_billed` to exclude only `vendor_refund` credits (which are
tracked in a separate `total_refunded` row) and include everything else:

```python
# CORRECT – includes vendor_bill, inventory_purchase, and any future types
total_billed = JournalLine.objects.active().filter(
    account=vendor.payable_account,
).exclude(
    journal_entry__transaction_type="vendor_refund",
).aggregate(v=Sum("credit"))["v"] or Decimal("0")
```

### Why `.exclude("vendor_refund")` instead of listing known types?
Listing types explicitly (e.g. `__in=["vendor_bill", "inventory_purchase"]`)
creates a hidden maintenance trap: every future transaction that posts a payable
credit must also be added to that list, or the detail view silently under-counts
again.  Excluding only the one type (`vendor_refund`) that is intentionally
tracked separately is far more future-proof.

## Correct Payable Balance Formula (Vendor Detail)

```
net_payable  = total_billed  -  total_paid  -  total_refunded
             = (all payable credits except vendor_refund)
               – (all payable debits)
               – (vendor_refund credits, shown as their own line)
```

`net_due` (shown as the final card in the UI) further subtracts any
outstanding advances:

```
net_due = net_payable – net_advance
```

## Pre-existing Inconsistency to Be Aware Of

`vendor.balance_payable()` (model method) does a simple
`sum(credit) – sum(debit)` on **all** journal lines of the payable account.
This **includes** `vendor_refund` credits as a liability increase, whereas the
detail view formula **subtracts** them (treating them as a reduction).

The two values will differ whenever there are vendor refunds:

| Scenario | `balance_payable()` | `net_payable` (detail view) |
|---|---|---|
| No refunds | ✅ identical | ✅ identical |
| With vendor refunds | ≠ | ≠ |

The detail view formula is the more **business-correct** one (a refund reduces
what we owe), so any future work that surfaces payable balances should prefer
the view's formula over the model's `balance_payable()` method, or the model
method should be updated to exclude refund credits from the sum.

## General Rules to Avoid Repeating This Bug

1. **Never filter journal lines by a single `transaction_type`** when the
   intent is "all entries that post to this account in this direction".  Use
   `.exclude()` for the one type you want to isolate, not `.filter()` for the
   one type you currently know about.

2. **When adding a new transaction type that credits a vendor's payable
   account**, check `vendors/views.py → VendorDetailView.get_context_data()`
   and `vendors/models.py → balance_payable()` to ensure the new type is
   properly counted.

3. **The vendor list and vendor detail must agree on the payable figure**
   (absent refunds). If they diverge, trace which query is excluding a
   transaction type.
